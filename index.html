git push 远端分之不成功

发表于 2018-10-10

git push 本地分之到码云时，总是提示不成功，如果报failed to push some refs to ‘https://gitee.com 这个错误,错误的主要原因是码云中的README.md文件不在本地代码目录中。

解决方法:

可以通过如下命令进行代码合并 (注：pull=fetch+merge)

git pull –rebase origin master

README.md在本地分之中，接下来在push 就成功了

Java异常

Exception 和 Error 的区别

  • Exception 和 Error 都是继承了 Throwable 类，在 Java 中只有 Throwable 类型的实例才可以被抛出（throw）或者捕（catch），它是异常处理机制的基本组成类型。

    • 简单描述：
    • 1.Error：编译时错误，系统错误，虚拟机出错，我们处理不了，也不需要我们来处理。
    • 2.Exception：运行时异常，可以捕获的异常，且作出处理。也就是要么捕获异常并作出处理，要么继续抛出异常。

  • 2.不要生吞异常。如果我们不把异常抛出，或者不输出到日志当中，无法诊断到底是哪里出现异常，以及产生异常的原因。

    • 我们从性能角度来审视一下 Java 的异常处理机制，这里有两个可能会相对昂贵的地方：
      • try-catch 代码段会产生额外的性能开销，或者换个角度说，它往往会影响 JVM 对代码进行优化，所以建议仅捕获有必要的代码段，尽量不要一个大的 try 包住整段的代码；与此同时，利用异常控制代码流程，也不是一个好主意，远比我们通常意义上的条件语句（if/else、switch）要低效。
      • Java 每实例化一个 Exception，都会对当时的栈进行快照，这是一个相对比较重的操作。如果发生的非常频繁，这个开销可就不能被忽略了。
      • 当我们的服务出现反应变慢、吞吐量下降的时候，检查发生最频繁Exception也是一种思路

    NoClassDefFoundError 和 ClassNotFoundException 有什么区别

  • NoClassDefFoundError是一个错误(Error)，而ClassNOtFoundException是一个异常，在Java中对于错误和异常的处理是不同的，我们可以从异常中恢复程序但却不应该尝试从错误中恢复程序。
  • ClassNotFoundException的产生原因：Java支持使用Class.forName方法来动态地加载类，任意一个类的类名如果被作为参数传递给这个方法都将导致该类被加载到JVM内存中，如果这个类在类路径中没有被找到，那么此时就会在运行时抛出ClassNotFoundException异常。
  • ClassNotFoundException的产生原因：Java支持使用Class.forName方法来动态地加载类，任意一个类的类名如果被作为参数传递给这个方法都将导致该类被加载到JVM内存中，如果这个类在类路径中没有被找到，那么此时就会在运行时抛出ClassNotFoundException异常。
  • NoClassDefFoundError产生的原因在于：如果JVM或者ClassLoader实例尝试加载（可以通过正常的方法调用，也可能是使用new来创建新的对象）类的时候却找不到类的定义。要查找的类在编译的时候是存在的，运行的时候却找不到了。这个时候就会导致NoClassDefFoundError.造成该问题的原因可能是打包过程漏掉了部分类，或者jar包出现损坏或者篡改。解决这个问题的办法是查找那些在开发期间存在于类路径下但在运行期间却不在类路径下的类。

finally块

  • finally块不管异常是否发生，只要对应的try执行了，则它一定也执行。只有一种方法让finally块不执行：System.exit()。因此finally块通常用来做资源释放操作：关闭文件，关闭数据库连接等等。
    • 在try块中打开资源，在finally块中清理释放这些资源。
    • finally块没有处理异常的能力。处理异常的只能是catch块。
    • 在同一try…catch…finally块中 ，如果try中抛出异常，且有匹配的catch块，则先执行catch块，再执行finally块。如果没有catch块匹配，则先执行finally，然后去外面的调用者中寻找合适的catch块。
    • 在同一try…catch…finally块中 ，try发生异常，且匹配的catch块中处理异常时也抛出异常，那么后面的finally也会执行：首先执行finally块，然后去外围调用者中寻找合适的catch块。

自定义异常

  • 如果要自定义异常类，则扩展Exception类即可，因此这样的自定义异常都属于检查异常（checked exception）。如果要自定义非检查异常，则扩展自RuntimeException。

    • 自定义的异常应该总是包含如下的构造函数：

      • 一个无参构造函数
      • 一个带有String参数的构造函数，并传递给父类的构造函数。
      • 一个带有String参数和Throwable参数，并都传递给父类构造函数。
      • 一个带有Throwable 参数的构造函数，并传递给父类的构造函数。

throw和throws

- throw用在方法体内。
- throws用在方法声明后面，表示再抛出异常，由该方法的调用者来处理。
- throw是具体向外抛异常的，抛出的是一个异常实例。
- throws声明了是哪种类型的异常，使它的调用者可以捕获这个异常。
- throw，如果执行了，那么一定是抛出了某种异常了，安生throws表示可能出现，但不一定。
- 同时出现的时候，throws出现在函数头、throw出现在函数体，两种不会由函数去处理，真正的处理由函数的上层调用处理。

异常的总结

  • 不要推诿或延迟处理异常，就地解决最好，并且需要实实在在的进行处理，而不是只捕捉，不动作。
  • 一个函数尽管抛出了多个异常，但是只有一个异常可被传播到调用端。最后被抛出的异常时唯一被调用端接收的异常，其他异常都会被吞没掩盖。如果调用端要知道造成失败的最初原因，程序之中就绝不能掩盖任何异常。
  • 不要在finally代码块中处理返回值。
  • 按照我们程序员的惯性认知：当遇到return语句的时候，执行函数会立刻返回。但是，在Java语言中，如果存在finally就会有例外。除了return语句，try代码块中的break或continue语句也可能使控制权进入finally代码块。
  • 请勿在try代码块中调用return、break或continue语句。万一无法避免，一定要确保finally的存在不会改变函数的返回值。
  • 函数返回值有两种类型：值类型与对象引用。对于对象引用，要特别小心，如果在finally代码块中对函数返回的对象成员属性进行了修改，即使不在finally块中显式调用return语句，这个修改也会作用于返回值上。
  • 勿将异常用于控制流。
  • 如无必要，勿用异常。

Git

什么是git

- git是去中心化的分布式版本控制系统，相较于svn，更方便的进行版本控制，代码管理，以及一些管理代码之外的事情，管理文档等。
- 想要更好的去了解Git，可以去Google上看官方文档，以下内容是我在工作当中的总结。

Git安装

  • git安装
    • Mac上安装Git ，在控制台输入 brew install git ，brew自行了解。
    • windows安装Git,直接在官网上下载，或者其他渠道下载，选择默认选项安装即可。安装完之后，鼠标右击会出现 Git bash 的选项，点击会弹出命令行窗口。
    • git –version 查看版本.

初始化本地库

  • 初始化 init:
    $ mkdir testGit首先创建一个用于存放初始化git的目录。
    cd testGit/进入目录。
    • 执行
      git init命令初始化本地仓库。
      Initialized empty Git repository in /Users/mumu/github/testGit/.git/创建完成。会看到目录下有一个.git文件夹，里面存放的便是一些版本信息。如下图：

      .git文件当中的所有信息。如果没有看到的话，文件是默认隐藏的，ls -ah便能看到啦。

  • add

      vim test.txt 之后我们要把它加入本地仓库当中
      git add test.txt这样就可以添加进来，如果添加所有文件 可以使用add . ,不用去指定文件名。
    • 添加完毕，执行
      git status查看仓库状态，出现如下图所示：

      添加文件成功！
    • 现在只是在仓库中添加了一个文件，git还不能对它进行版本管理，如果想让git能对它进行管理的话，必须提交到本地仓库
      git commit -m"提交的信息"这样我们就把添加的文件提交到版本库啦，如下图：
    • 上面两步就把文件成功的提交到了本地仓库当中。

本地库创建完成，我们还得将代码push到远程仓库，和远程仓库关联的有两种方式。
  • clone
    • clone 命令是将在远端创建的库克隆到本地，这样就不用再本地初始化。

  • remote add origin
    • 如下图所示，首先在本地创建文件夹，初始化本地库，之后git remote add origin xxxx

  • 最好的方式是先创建远程仓库在clone到本地，之后分支是本地创建推送到远端。

添加用户

  • 如果每个人都获得你的项目地址，push的时候，远程仓库是要验证个人信息的。
    $ git config –global user.name “Your Name”
    $ git config –global user.email “email@example.com“
    在终端输入个人信息，git 会保存在本地git信息中，也可以生秘钥.

推送和拉取

  • push
    • 只要本地commit代码之后，就可以使用：
      git push origin master
      来推送本地代码到远端了，push 推，origin 远程仓库的名字，master 分支。

  • pull

    • 上面有clone 把远端代码克隆到本地，pull命令是把分支代码拉取到本地
      $ git pull origin master

  • 注意：当多人合作开发时，应先pull，在本地解决冲突之后，push到远端.

版本回退

  • git log
    • 使用
      git log 命令，可以看到提交的版本信息.信息如下图：

      0fb56d72f8ee66e759d8f762b66f8fc313cf660f 为版本号，没提交一次生成一条新的hard。
  • git reset
    • git reset –hard [hard] 加上版本号，就能回到这个版本好提交的版本。
  • git reflog
    • 这是获得所有提交过的版本好。回到过去之后，还可以回到未来

分支 Branch

  • 每个git仓库都会有很多分支，比如在开发项目时，master就是主分支，还有测试分支，bug分支等，一般远程仓库就只有master和dev分支，其他分支都是在自己的本地，修改好代码之后合并到其他分支上。

    • git branch -a 时查看所有本地分支和远程分支。

    • git checkout branch 切换分支 ，比如当前在master分支下 ，要切换到dev分支下git checkout dev 就会切换到dev下，切换之前必须提交。

    • 创建分支。
      git branch newbranch 会创建一个新的分支，只会在本地创建，不会在远端创建，push时指定远端分支。

合并 merge

  • 当需要在dev分支下需改一个bug的时候，不能动dev分支的代码，需要创建一个新的分支，在bug分支下修改，修改之后需要把代码移植到dev，这时，就用到了合并 merge。
    • 首先切换到dev分支 ，
      git merge bug 这样就会把bug分支合并到dev分支上。

冲突

  • 当合并分支，或者是从远端pull代码的时候，由于是多人协作开发，肯定有些地方被别人修改过，这样到本地就会出现冲突，打开冲突的文件，手动修改，解决之后，就可以提交了。

还原

  • 本地修改代码之后 ，如果还没有提交，但是又想回到修改之前的代码，可以使用
    git revert 还原到修改之前的代码。

其它

  • git的一些其他命令 git –help 中都有指出，其他命令详见。在使用过程当中，也可以使用图形化管理工具，更便于操作。

git版本回到未来和回到过去

发表于 2016-10-09

Git的版本回退，先获得版本号git log ，之后根据hard 回退版本，git reset –hard 版本号，这样就能回到过去。

Git的回到未来，获得版本号git reflog ,根据在未来commit测信息，获得hard，之后git reset –hard 版本号 ，这样就可以回到未来。
